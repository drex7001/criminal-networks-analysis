# Spec 07 — UI: the Investigation Workspace

Status: active (rewritten 2026-07-18 — ADR-032: one durable React + TypeScript
workspace from Phase 2; the three-stage legacy-explorer plan is superseded) ·
GOAL.md §29–30

## 1. Strategy — one app, grown in place

There is **one** UI: `ui/` — React 18 + TypeScript + Vite, started in Phase 2
and grown through every later phase. No interim server-rendered stack, no
throwaway panels, no parallel UI generations (ADR-023's replace-never-extend,
applied to our own work). The legacy Cytoscape explorer and its anonymous
`/api/*` surface are deleted in Phase 2 when the workspace's graph view lands
(ADR-026).

| Phase | The same app gains |
|---|---|
| **P2** | Auth shell (Keycloak OIDC PKCE), source landing/extraction status, review queue (typed suggestions), identity adjudication, graph view + provenance panel, entity search |
| **P4** | Object views (entity-360), cases, hypotheses, tasks/leads, timeline + as-of mode, case-centric layout |
| **P5** | Map view (MapLibre) synced with timeline/graph |
| **P6** | Global search, object-set builder, findings panel, watchlist triage |
| **P7** | Redaction preview, compartment UX, break-glass flow |
| **P8** | Assistant surfaces (citation-first) |

## 2. Technical shape (P2 — ADR-032)

- **Auth:** `oidc-client-ts` / `react-oidc-context` (Article XII — no
  hand-rolled OIDC state machine). Tokens held in memory, never localStorage;
  silent renew per the library; logout revokes; CSP + security headers served
  with the app; idle/absolute session timeouts configured in Keycloak.
- **API client:** generated from the FastAPI OpenAPI document (stable
  operation IDs are an API convention from P2). The P3 ontology-aware client
  (`sdk/ts/`) replaces it types-first, with no screen rewrites.
- **Data:** TanStack Query (or equivalent) for fetching/caching; no global
  state framework until a real need appears.
- **Graph canvas:** Cytoscape.js inside React (exists, works; Sigma.js only
  behind the plan §2 WebGL trigger).
- **Quality bar:** function over polish in P2 — cosmetic work is P4's;
  type-check + build + one demo-loop e2e in CI.

## 3. Ontology-driven screens (Article XI — P4)

Generic components render from `ui_meta.json` (generated from the ontology):

- **Entity page**: title/subtitle from `display`; claims grouped by predicate;
  each claim row shows assertion type, grading chips (reliability /
  credibility / verification — Article III), validity window, source link,
  contradiction badge.
- **Claim detail**: full provenance — excerpt, source record, derivative chain
  down to the original file; relations; retraction state.
- **Forms**: create/edit claim drafts built from ontology property/predicate
  specs and action parameter schemas (P3) — adding `vessel` to a domain module
  yields a working vessel page with no new UI code. Evaluate
  JSON-Schema-driven form generation (e.g. RJSF) before hand-building (H-20);
  bespoke code is reserved for provenance, conflict display, and governed
  actions — the surfaces that *are* the product.

## 5. Interaction principles (binding — GOAL.md §30)

- Search first, expand second; expansion always bounded (hops/types/time/confidence).
- Every claim opens its source; every metric shows its caveat (Article IX).
- Suggested (machine) material is visually distinct from recorded claims —
  different styling, separate legend entry; never mixed silently into the
  canonical view (Article VII).
- Contradictions are a visible lens, not a footnote (Article VIII).
- Current vs historical state clearly separated; as-of mode has a persistent
  banner stating exactly what the snapshot holds constant (B-11).
- No red-means-criminal styling; categories use the ontology palette.
- Restricted data is absent, not teased (no "N hidden" counts — specs/03 §4;
  marked-redaction and counts modes are P7 policy decisions, H-25).
- Retained from the prototype (as *requirements*, not code): layer/category
  toggles, confidence filter, temporal slider with play, cell coloring,
  source-cited detail cards.
